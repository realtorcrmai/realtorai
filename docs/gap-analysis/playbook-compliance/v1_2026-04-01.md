## NEW finding from Pass 3: The enforcement model is fundamentally user-overridable

**The hooks output "BLOCKED" but the user can approve anyway.** This means:
- playbook-gate.sh blocks edits → user approves → edit proceeds
- completion-gate.sh blocks completion → user approves → task ends without compliance

This is BY DESIGN in Claude Code — hooks are advisors, not absolute blockers. But it means the AGENT must be the enforcer, not the hook. The hook is a reminder; the agent must refuse to proceed even if the user approves.

**This changes the entire enforcement model.** Hooks are Layer 2 (gate) but they're actually Layer 1 (reminder) in practice. The real enforcement must be in:
- Layer 3 (agent self-check) — agent must check deliverables
- Agent memory (feedback entries) — agent must remember to create docs

## What Pass 3 found that Passes 1-2 missed

1. **Stale task file is worse than no task file** — it gives a false signal that a task is active, but with wrong phase structure. The gate blocks legitimately because phases.classified is undefined.

2. **User override defeats hook enforcement** — hooks are soft blocks, not hard blocks. The agent must be the enforcer.

3. **The playbook assumes hooks are hard blocks** — Section 15 says "Layer 2 blocks edits" but in reality it's "Layer 2 SUGGESTS blocking, user decides."

4. **No deliverable enforcement exists at ANY layer** — not hooks, not L3 gate, not Quick Reference, not completion-gate. It's entirely prose in Section 1.3.

**Root cause 4: The completion-gate hook doesn't check for deliverables**

It checks: tsc passes + compliance_logged flag. It doesn't check:

### What "no surface-level" means for this analysis

Pass 1-3 identified WHAT is missing. Pass 4 explains WHY — the playbook puts deliverables at the wrong point in the flow, the Quick Reference omits them entirely, and no checkpoint forces the agent to think about them.

The fix isn't "add more rules" — it's "put deliverables where the agent actually looks."
# Playbook Compliance Gap — Pass 5: Completeness

## Re-reading Pass 4... what else is missing from the entire analysis?

Pass 4 identified WHY deliverables are skipped (wrong position in flow, missing from Quick Reference, missing from L3 gate). But:

### MISS 1: The stale current-task.json needs to be cleaned up
Nobody addressed this — it's causing the gate to fire on every edit with a misleading block message. This is an immediate fix needed before anything else.

### MISS 2: The playbook says tests should exist at tests/<feature>.md but the actual test infrastructure is elsewhere
The test files are:
- tests/browser/ — Playwright specs
- src/__tests__/ — Vitest unit tests
- scripts/eval-*.mjs — Eval scripts
- scripts/test-suite.sh — Integration tests

NONE of these follow the tests/<feature>.md pattern. The playbook's test doc requirement is disconnected from where tests actually live. This is a documentation mismatch, not just a missing deliverable.

### MISS 3: No analysis of which deliverables are ACTUALLY valuable vs checkbox items
Not all deliverables are equal:
- usecases/<feature>.md → HIGH VALUE if kept updated. Enables new devs to understand features.
- tests/<feature>.md → LOW VALUE as separate doc. Tests should be in test files, not markdown.
- Compliance log → MEDIUM VALUE for audit trail. Auto-logging is the right approach.
- Running tests → HIGH VALUE. The most important deliverable that's skipped.

The playbook mandates all equally, which means all get skipped equally. Better: mandate the high-value ones, make the low-value ones optional.

### MISS 4: No analysis of how much time deliverables add
For a Medium task (2-4 hours of coding):
- Creating usecases/<feature>.md: +30 min
- Creating tests/<feature>.md: +20 min (if separate from actual tests)
- Running tests: +5-30 min (depending on scope)
- Compliance log: +2 min (auto-logged)
- Total overhead: ~1 hour (+25-50% of task time)

This overhead is why agents skip it — the "implement" phase feels productive, the "document" phase feels like paperwork. The playbook doesn't acknowledge this tradeoff.

### MISS 5: The playbook doesn't distinguish between "create new" and "update existing" deliverables
For a new feature, creating usecases/<feature>.md makes sense.
For a bug fix or refactor, updating an existing usecase doc makes sense.
For a documentation change, no usecase doc is needed.

The playbook says "every significant feature must have" but doesn't define "significant" or differentiate by task type.

## What Pass 6 reveals that previous passes missed:

The user's workflow is a **pipeline**: Gap → Implement → Use Case → Test Case → Test. Each step feeds the next. But in practice, we treat them as independent steps and skip the ones without enforcement.

**The fundamental problem:** The playbook describes a pipeline but enforces only 2 of 5 steps (gap analysis and implementation). The other 3 (use case, test case, testing) are prose requirements with zero enforcement.

**What would fix it:** Make steps 3-5 BLOCKING in the same way steps 1-2 are:
- Use case doc: Required before PR (like classification is required before edits)
- Test case doc: Required before PR (or merged with actual test files)
- Test execution: Required before PR (completion-gate should run tests, not just tsc)
