## CRITICAL ISSUES (Fix Immediately)

### C1: RAG Chat Widget is Orphaned
- **Evidence:** `grep -r "ChatWidget" src/app/` returns ZERO results
- **Impact:** Entire RAG system (15 lib files, 10 API endpoints, 8 components) invisible to users
- **Fix:** Add `<ChatWidget />` to `src/app/(dashboard)/layout.tsx`
- **Effort:** 5 minutes

### C2: 10 Duplicate Migration Numbers
- **Evidence:** Two files each for numbers 005, 050, 051, 052, 053, 054, 055, 058, 062, 063
- **Impact:** Migration runner may skip one of each pair, causing schema gaps
- **Fix:** Rename duplicates to sequential 066-075
- **Effort:** 30 minutes

### C3: RLS Policies Use auth.role() Not auth.uid()
- **Evidence:** Migration 065 creates policies with `USING (auth.role() = 'authenticated')`
- **Impact:** Any authenticated user can access any realtor's data (multi-tenant isolation broken at DB level)
- **Fix:** Change to `USING (realtor_id = auth.uid()::uuid)` or `USING (realtor_id = (auth.jwt()->>'sub')::uuid)`
- **Effort:** 2 hours

### C4: Social Media Cron Not Scheduled
- **Evidence:** `/api/cron/social-publish` route exists but NOT in vercel.json cron config
- **Impact:** Scheduled social posts never auto-publish
- **Fix:** Add to vercel.json
- **Effort:** 5 minutes

## ARCHITECTURAL PRINCIPLES ASSESSMENT

Evaluated against 4 foundational principles. Each verified by reading actual code.

### A1: UI Design — Minimal Keyboard, Touch-Friendly (7/10)

**What exists (verified):**
- `MobileNav.tsx`: 44px min touch targets (`min-w-[44px] min-h-[44px]`), fixed bottom bar
- Responsive breakpoints: `p-4 md:p-6 lg:p-8` across all detail pages
- `MobileDetailSheet.tsx` + `MobileSidebarSheet.tsx`: sheet-based mobile panels
- `globals.css`: professional design system with 40+ custom properties, dark/light theme, 5 elevation levels
- `manifest.json`: PWA-ready configuration

**What's missing:**
- No keyboard accessibility (no tab order, no autofocus, no Enter-to-submit on forms)
- No ARIA labels on interactive elements (buttons have text only)
- No swipe/gesture support for mobile navigation
- No accessibility testing (axe-core tests exist but found 17 violations)
- No formal UI design guidelines document

### A2: Data Model & Master Data Model (8/10)

**What exists (verified):**
- 65 migrations, 50+ tables with FK constraints, CHECK constraints, UNIQUE constraints
- `src/types/database.ts`: 1278 lines of comprehensive type definitions
- JSONB columns for flexible data: `buyer_preferences`, `seller_preferences`, `forms_status`, `enrichment_status`
- Audit trail: `created_at`/`updated_at` on all tables with trigger-based auto-update
- Multi-tenant: `realtor_id` on 40+ tables with indexes and immutability trigger
- Indexes on all FK columns, status/type columns, temporal columns

**What's missing:**
- No ERD or visual schema diagram
- No data dictionary documenting fields and business meaning
- No documented schemas for JSONB columns (freeform risk)
- No soft-delete pattern (hard deletes cascade)
- No master reference data tables (statuses/types are inline enums)
- 10 duplicate migration numbers (schema integrity risk)

### A3: Persona Mapping & Workflow Definition (7/10)

**What exists (verified):**
- `docs/user-journeys.md`: 5 personas (Prospect, Buyer, Seller, Other Agents, Realtor)
- `src/lib/auth.ts`: 2 roles (admin, realtor) with feature flag array
- `src/lib/features.ts`: 14 features independently toggleable
- 8-phase listing workflow in `ListingWorkflow.tsx` (40.9KB)
- Contact journey engine: auto-enrollment, phase advancement, trust levels (0-3)
- Engagement scoring: 12 click categories, graduated escalation (AI→digest→alert→call→urgent)
- 6 contact types: buyer, seller, customer, agent, partner, other

**What's missing:**
- No formal persona profiles (demographics, pain points, goals)
- Only 2 user roles — no agent/manager/admin tiers, no team support
- Buyer-agent stored as flat text, not modeled as contact relationship
- Workflow state machine implicit (sequential integers), not formally documented
- No documented phase entry/exit conditions
- No compliance gates between phases (e.g., FINTRAC before MLS submission)

### A4: Phase One Prototyping & Rollout Plan (6/10)

**What exists (verified):**
- 14 features independently toggleable via `enabledFeatures` session array
- Domain-separated actions (32 files) and API routes (130+)
- Tenant isolation wrapper (`getAuthenticatedTenantClient()`)
- Git workflow: feature branch → dev PR → main PR (documented)
- Deploy skill: `.claude/skills/deploy.md` with service architecture
- Test infrastructure: Vitest + Playwright + 10 eval scripts

**What's missing:**
- No staged rollout plan (no beta users, no gradual % rollout)
- No A/B testing or feature experiments infrastructure
- No canary/blue-green deployment strategy
- No incident runbook or rollback procedures
- Test coverage metrics unknown (no thresholds)
- No monitoring/observability (no Sentry, no dashboards)
- No performance benchmarks or load testing
- No API versioning strategy
