## WHY THE PLAYBOOK KEEPS FAILING

Despite being 1684 lines with detailed rules, the playbook fails repeatedly. Here's why, based on evidence from this session:

### Problem 1: The playbook is followed 0% of the time by AI agents

**Evidence from this session alone:**
- Gap analysis launched without pipeline → caught by user
- 22 RAG features marked "FIXED" without verifying a single one against code
- 67 gaps marked "done" based on agent reports, not code verification
- Architectural principles analysis launched without classification
- Multiple "let me fix it" cycles without following classify → scope → plan

**Root cause:** The playbook is 1684 lines. An AI agent can't hold all 82 sections in working memory. By the time it reads Section 4 (task playbooks), it's forgotten Section 2 (pre-flight). The length itself defeats compliance.

**The playbook tells you WHAT to do but doesn't make it EASY to do.** A 10-line checklist would be followed more reliably than 1684 lines of prose.

### Problem 2: Hooks exist but don't enforce enough

**What hooks enforce:**
- `playbook-gate.sh`: Blocks edits without `current-task.json` → BUT agents can skip creating the file and go straight to Bash/Read instead
- `completion-gate.sh`: Runs tsc on Stop → Works, but only catches compile errors, not process violations
- `git-protection.sh`: Blocks force push → Works

**What hooks DON'T enforce:**
- Running the agent-pipeline first (no hook for this)
- Outputting a classification block before any tool call
- Writing scope paragraph before coding (Medium+)
- Verifying against actual code during gap analysis
- Following the 5-pass process for analysis tasks

### Problem 3: No consequences for violation

The playbook says "work that skips the playbook WILL be reverted" but there's no mechanism to detect or revert. The compliance log is supposed to be mandatory but is always empty because no one fills it in.

## MISSING SECTIONS (from recent learnings)

### M1: 5-Pass Analysis Process ❌ NOT IN PLAYBOOK

The v8 playbook we wrote earlier had this as a primary section. The current expanded playbook lost it entirely. This is critical — every gap analysis, PRD, and architecture doc should go through 5 iterative passes:

1. Pass 1: Find all gaps yourself
2. Pass 2: Compare with best-in-market
3. Pass 3: **Verify claims against actual code** (not reports)
4. Pass 4: Check depth, expand surface-level sections
5. Pass 5: Final read, mark as final

### M2: GAP_ANALYSIS Task Type ❌ NOT IN PLAYBOOK

There are 14 task types but no GAP_ANALYSIS type. We do this frequently. Rules needed:
- MUST read actual source code, not previous reports
- MUST verify every claim with file:line references
- MUST save as versioned file in `docs/gap-analysis/`
- "Code written" ≠ "Feature works" — check integration, not just file existence

### M3: Architectural Principles Section ❌ NOT IN PLAYBOOK

The 4 principles from the meeting notes apply to every feature we build:

1. **UI Design:** Touch-friendly, minimal keyboard, WCAG accessibility, ARIA labels
2. **Data Model:** ERD required, data dictionary, documented JSONB schemas
3. **Persona Mapping:** Role matrix, workflow state machines, documented entry/exit conditions
4. **Rollout Plan:** Staged deployment, monitoring, incident runbook, performance benchmarks

### M4: Code Verification Rule ❌ NOT ENFORCED

Memory says "Gap analysis must verify against actual code, not just docs/reports" but the playbook has zero mention of this. It should be a Hard Constraint:

> **HC-13: All analysis, gap reports, and status claims MUST be verified by reading actual source code.** Never trust previous reports, agent outputs, or documentation alone. "Code written" ≠ "Feature works."

### M5: Playbook Brevity Problem ❌ NOT ADDRESSED

At 1684 lines, the playbook is too long for any agent to follow consistently. The Quick Reference Card (Section 10) is 80 lines and more effective. Consider:
- Core playbook: <300 lines (hard constraints + SDLC + quick reference)
- Extended reference: separate files for task playbooks, model chaining, governance

## HONEST VERDICT

The playbook is **comprehensive but not effective**. It documents everything that should happen but lacks the mechanisms to make it happen. The core issue: **length defeats compliance**. An AI agent reading 1684 lines of rules will still skip steps because the rules aren't at the point of action — they're in a separate document that gets forgotten mid-task.

**What would actually work:**
1. A 200-line core playbook with only hard constraints + SDLC + quick reference
2. Hooks that enforce the pipeline and classification (not just edit-blocking)
3. A GAP_ANALYSIS task type that mandates code verification
4. The 5-pass process baked into DESIGN_SPEC and GAP_ANALYSIS task types
