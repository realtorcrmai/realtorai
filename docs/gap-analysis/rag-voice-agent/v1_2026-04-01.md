## Pass 4: Depth Check — Root Cause Analysis

### Why These Gaps Exist (not just "what's missing")

**G1-G3 (Tenant isolation in RAG lib):** Migration 070 added realtor_id to DB tables and RLS policies, but the RAG library functions (`mention-parser.ts`, `instant-search.ts`) were written before multi-tenancy was enforced. They use raw `.ilike()` queries that bypass RLS because the admin client is used for ingestion. **Root cause:** These utility functions were never updated when migration 070 was applied. The migration fixed DB-level RLS but code-level queries in these files still use unscoped patterns.

**G4 (Retriever has no fallback):** `fallback.ts` exists with `fullTextSearch()` but `retriever.ts:searchEmbeddings()` throws on error without invoking it. The caller (`chat/route.ts`) catches at top level but returns a generic error instead of falling back to FTS. **Root cause:** `fallback.ts` was written as a separate module but never wired into the primary retrieval path. The circuit breaker pattern was designed but not connected.

**G5 (Weak guardrails):** RAG has 5 patterns for domain-specific topics (tax, legal, financial). Agent adds 10 more (FINTRAC PII + injection). But both miss: "act as [role]", "ignore previous context", Unicode trick attacks, delimiter injection, and output manipulation attacks. **Root cause:** Guardrails were designed for topical safety (don't give tax advice) not adversarial safety (prompt injection). The threat model is incomplete — it protects against innocent misuse but not active attacks.

**G6 (No evaluation):** Neither system has automated quality metrics. The email marketing engine has a 7-dimension quality pipeline (`quality-pipeline.ts`) scoring relevance, tone, compliance, etc. — but this pattern was never applied to RAG or Voice. **Root cause:** RAG was built feature-first (make it work) without a quality gate. The email engine was built spec-first (PRD specified quality scoring). No PRD defined RAG quality metrics.

**G7 (Single shared voice API key):** `voice-agent-auth.ts` has a TODO for per-tenant keys. Currently one key authenticates all Voice Agent requests. **Root cause:** The Voice Agent was developed as a single-tenant prototype and multi-tenancy was added later. Auth was the last piece and the per-tenant key generation wasn't prioritized.

**G8 (No TTS caching):** Edge TTS generates audio deterministically for the same input. Repeated phrases ("I've created the task", "Looking up that contact") regenerate audio every time. **Root cause:** The voice pipeline was designed for correctness first. Caching was deferred as an optimization. No profiling was done to identify repeated phrases.

## Pass 7: Implementation Sanity — Challenging My Own Recommendations

**Challenge 1: "Are the CRITICAL gaps really critical?"**
G1 and G2 (tenant isolation in mention-parser and instant-search) — YES, these are cross-tenant data leaks. Any authenticated user can search all contacts/listings across all tenants. This is not theoretical — the ilike queries return all rows because the admin client bypasses RLS for these specific functions.

**Challenge 2: "Is expanding guardrails to 30+ patterns realistic?"**
Regex-based guardrails have diminishing returns. At 30+ patterns, maintenance becomes a burden and false positives increase. A better approach: keep current patterns for domain safety (tax/legal), add a classifier-based approach for prompt injection (e.g., LlamaGuard or a fine-tuned Haiku classifier). This is more maintainable and catches creative bypass attempts that regex misses. **Updated recommendation: 15-20 regex patterns + classifier for injection.**

**Challenge 3: "Is building a RAGAS-style evaluation pipeline worth 3 days?"**
Without evaluation, we can't know if changes improve or degrade quality. The email marketing engine proved this — the 7-dimension quality pipeline catches bad content before it reaches users. For RAG, even a minimal eval (20 golden questions, measure faithfulness + relevance) gives us a regression gate. **Recommendation stands, but start with 20 golden questions, not full RAGAS.**

**Challenge 4: "Per-tenant API keys — is this overengineering?"**
If only one realtor uses the system, a single key is fine. But the system is designed for multi-tenancy. One compromised key = all tenants exposed. Since we already have `tenant_api_keys` table pattern in the TODO, the effort is low. **Recommendation stands.**

**Challenge 5: "TTS caching — is the latency gain worth it?"**
Edge TTS is 1-3s per call. If a user asks 10 questions per session, and 3 responses reuse common phrases (confirmations, errors), caching saves ~3-9s total per session. This is measurable but not critical. **Downgrade from SHOULD to COULD.**

**Challenge 6: "Should we invest in reranking before fixing guardrails?"**
No. Security (guardrails, tenant isolation) before quality (reranking). A RAG system that gives great answers but leaks data across tenants is worse than one that gives okay answers securely. **Priority order confirmed: Security → Evaluation → Quality → Performance.**
