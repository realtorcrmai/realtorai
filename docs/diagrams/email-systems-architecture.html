Magnate 360 — Email Systems Architecture

6 independent email-sending systems with no shared frequency caps, 4 different send history tables, and no priority resolution. A new contact can receive 5+ emails on day one.

System 2 — Event Pipeline
2

Event Pipeline

CRM emits → Render processes
Sends to: Affected contacts (context-dependent per event)
⚡ TRIGGERS (5 wired)
  • listing_price_dropped: Realtor changes list price downward
  • listing_sold: Listing status changed to "sold"
  • showing_confirmed: Seller confirms showing via SMS YES
  • contact_birthday: Daily 8AM cron checks contact_important_dates
  • listing_matched_search: New listing matches buyer saved search prefs
  • ~12 more event types defined but not wired to pipelines
⏰ WHEN EMAILS SEND
  • Real-time events: Within 10 seconds (poll interval) of CRM action
  • Birthday: Daily at 8:00 AM Vancouver time
  • Saved search: Within 10s of new listing matching criteria
  • Rule caps: Min 24h between same event type (configurable per rule)
  • Weekly cap: Max 3 emails/week of same type per contact
  • Send mode: "auto" (instant) or "review" (queued for approval)
👥 RECIPIENTS
  • listing_price_dropped: All buyers who viewed/saved that listing
  • listing_sold: Seller + interested buyers (just_sold celebration)
  • showing_confirmed: Buyer agent who requested the showing
  • contact_birthday: The birthday contact (personal greeting)
  • listing_matched_search: Buyer whose saved search criteria match

To: sarah.chen@email.com • Type: editorial broadcast • Stores in: editorial_editions
Result: 5 emails in 2 minutes, 6th email within 24 hours

Each system's rate limiter says "I only sent 1 email" — none sees the other 4-5. Sarah's inbox: 3 welcome emails + 1 listing alert + 1 agent message + 1 editorial.

No Shared Frequency Cap

3 independent rate limiters (sendGovernor, resolveRuleForEvent, agent cap) each query different tables. No single query counts all sends to a contact.

4 Send History Tables

newsletters, editorial_editions, communications, agent_drafts. A global "what did we send this contact today?" query is impossible.

Welcome Email x3

Journey (System 1), Drip (System 5), and optionally Workflow (System 4) all send welcome emails on contact creation. Same content, 3 separate code paths.

No Priority Resolution

When 3 systems want to email the same contact, first-to-fire wins. No concept of "this listing alert is more important than a drip nurture."

Events Don't Drive Journeys

A showing_confirmed event doesn't advance buyer from lead → active. listing_sold doesn't move seller to past_client. Phase changes are time-based or manual only.

Workflow Writes to Wrong Table

Workflow emails write to communications, but its own rate limiter (sendGovernor) checks newsletters. Its sends are invisible to its own cap.

Target: Unified Send Gateway

All 4 surviving producers → 1 gateway → Resend. System 5 (Drip) absorbed into System 1 (Journey).
